## 数据流图(DFD)答题技巧

### DFD基本要素
**数据流**
- 表示：带箭头的直线
- 命名：使用名词或名词短语
- 规则：必须有起点和终点，不能直接连接外部实体

**加工(处理)**
- 表示：圆形或椭圆形
- 编号：按层次编号(1, 2, 3或1.1, 1.2, 1.3)
- 命名：使用动宾短语

**数据存储**
- 表示：开口矩形
- 编号：D1, D2, D3...
- 命名：使用名词

**外部实体**
- 表示：矩形框
- 特点：系统边界外的实体
- 命名：使用名词

### 绘制步骤和技巧

#### 第一步：识别外部实体
**识别方法**
- 找出与系统交互的人、组织或其他系统
- 通常在题目描述的开头和结尾出现
- 关键词：用户、管理员、客户、供应商、其他系统

**常见外部实体**
```
学生管理系统：学生、教师、管理员
图书管理系统：读者、图书管理员、采购员
销售系统：客户、销售员、仓库管理员
```

#### 第二步：识别数据存储
**识别方法**
- 找出系统需要保存的数据
- 通常对应数据库中的表
- 关键词：文件、数据库、记录、信息

**命名规范**
```
D1：学生信息
D2：课程信息
D3：选课记录
D4：成绩信息
```

#### 第三步：识别加工
**识别方法**
- 找出系统的主要功能
- 每个加工对应一个业务处理过程
- 关键词：处理、管理、计算、统计、查询

**编号和命名**
```
1. 学生注册
2. 课程管理
3. 选课处理
4. 成绩管理
5. 统计查询
```

#### 第四步：连接数据流
**连接规则**
- 外部实体 → 加工：输入数据
- 加工 → 外部实体：输出数据
- 加工 → 数据存储：存储数据
- 数据存储 → 加工：读取数据
- 加工 → 加工：中间数据传递

**数据流命名技巧**
```
输入类：学生信息、课程信息、选课申请
输出类：成绩单、课程表、统计报表
查询类：学生查询、成绩查询
确认类：注册确认、选课确认
```

### 常见错误及避免方法

**错误1：数据流直接连接外部实体**
```
错误：学生 → 教师 (数据流)
正确：学生 → 1.提交申请 → 教师
```

**错误2：数据存储之间直接连接**
```
错误：D1学生信息 → D2课程信息
正确：D1学生信息 → 2.处理 → D2课程信息
```

**错误3：加工没有输入或输出**
```
错误：孤立的加工
正确：每个加工至少有一个输入和一个输出
```

### 标准答题模板

#### 顶层DFD模板
```
外部实体1 → 系统 → 外部实体2
外部实体3 → 系统 → 外部实体4
```

#### 0层DFD模板
```
1. 确定外部实体(3-5个)
2. 确定主要加工(4-6个)
3. 确定数据存储(3-5个)
4. 连接数据流，确保平衡
```

## 数据库设计答题技巧

### ER图设计

#### 实体识别技巧
**识别方法**
- 找出题目中的名词
- 判断是否是独立存在的事物
- 通常对应数据库中的表

**常见实体类型**
```
人员类：学生、教师、员工、客户
物品类：商品、设备、图书
事件类：订单、课程、项目
概念类：部门、课程、类别
```

#### 属性识别技巧
**属性类型**
- **标识属性**：唯一标识实体(主键)
- **描述属性**：描述实体特征
- **联系属性**：表示实体间关系的属性

**命名规范**
```
学生实体：学号(主键)、姓名、性别、年龄、专业
课程实体：课程号(主键)、课程名、学分、学时
选课关系：学号、课程号、成绩、选课时间
```

#### 联系识别技巧
**联系类型判断**
- **1:1关系**：一个实体对应另一个实体的一个实例
- **1:N关系**：一个实体对应另一个实体的多个实例
- **M:N关系**：两个实体都可以对应多个实例

**常见联系模式**
```
学生-专业：N:1 (多个学生属于一个专业)
学生-课程：M:N (学生可选多门课程，课程可被多个学生选)
部门-经理：1:1 (一个部门有一个经理)
```

### 关系模式转换

#### ER图转关系模式规则
**实体转换**
```
实体E(A1, A2, ..., An) → 关系R(A1, A2, ..., An)
主键保持不变
```

**联系转换**
```
1:1联系：合并到任一实体关系中
1:N联系：在N端加入1端的主键作为外键
M:N联系：单独建立关系表
```

#### 标准转换模板
```
学生(学号, 姓名, 性别, 年龄, 专业号)
专业(专业号, 专业名, 所属学院)
课程(课程号, 课程名, 学分, 学时)
选课(学号, 课程号, 成绩, 选课时间)

外键约束：
学生.专业号 → 专业.专业号
选课.学号 → 学生.学号
选课.课程号 → 课程.课程号
```

### 规范化处理

#### 规范化步骤
1. **检查1NF**：确保属性原子性
2. **检查2NF**：消除部分函数依赖
3. **检查3NF**：消除传递函数依赖
4. **检查BCNF**：消除主属性依赖

#### 答题格式
```
原关系：R(A, B, C, D, E)
函数依赖：AB → C, A → D, D → E

违反2NF：A → D (部分依赖)
违反3NF：A → D → E (传递依赖)

分解为3NF：
R1(A, B, C)  主键：AB
R2(A, D)     主键：A
R3(D, E)     主键：D
```

## UML图设计答题技巧

### 用例图设计

#### 参与者识别
**识别方法**
- 找出与系统交互的角色
- 区分主要参与者和次要参与者
- 考虑系统管理员角色

**常见参与者**
```
用户类：普通用户、VIP用户、游客
管理类：系统管理员、业务管理员
系统类：外部系统、定时器、数据库
```

#### 用例识别
**识别方法**
- 从参与者的角度思考系统功能
- 每个用例代表一个完整的业务流程
- 用动宾短语命名

**用例分类**
```
基本用例：登录、注册、查询、修改
业务用例：下订单、支付、发货、退款
管理用例：用户管理、权限管理、系统配置
```

#### 关系处理
**包含关系(include)**
```
用途：提取公共功能
示例：下订单 include 验证用户身份
```

**扩展关系(extend)**
```
用途：处理异常或可选功能
示例：登录 extend 忘记密码
```

### 类图设计

#### 类的识别
**识别方法**
- 从需求中提取名词
- 分析对象的职责
- 考虑设计模式的应用

**类的分类**
```
实体类：对应业务实体(User, Product, Order)
控制类：处理业务逻辑(UserController, OrderService)
边界类：处理界面交互(UserView, OrderForm)
```

#### 关系设计
**关联关系**
```
表示：实线
多重性：1, 0..1, 1..*, 0..*, n..m
角色名：关联端的名称
```

**继承关系**
```
表示：实线三角箭头
用途：is-a关系
示例：Student → Person
```

**依赖关系**
```
表示：虚线箭头
用途：使用关系
示例：Controller → Service
```

### 时序图设计

#### 对象识别
**对象类型**
- 边界对象：用户界面
- 控制对象：业务逻辑
- 实体对象：数据对象

#### 消息设计
**消息类型**
```
同步消息：实线箭头(方法调用)
异步消息：开放箭头(事件通知)
返回消息：虚线箭头(返回值)
自调用：指向自己的箭头
```

**消息命名**
```
格式：方法名(参数): 返回类型
示例：login(username, password): boolean
```

## 面向对象设计答题技巧

### 设计模式应用

#### 常考设计模式
**单例模式**
```
应用场景：配置管理、日志记录、数据库连接
实现要点：私有构造函数、静态实例、全局访问点
```

**工厂模式**
```
应用场景：对象创建、产品族管理
实现要点：抽象工厂、具体工厂、产品接口
```

**观察者模式**
```
应用场景：事件处理、模型-视图分离
实现要点：主题接口、观察者接口、通知机制
```

#### 答题格式
```
1. 识别设计模式类型
2. 说明应用场景和优点
3. 画出类图结构
4. 说明关键实现要点
```

### SOLID原则应用

#### 单一职责原则(SRP)
```
问题：类承担多个职责
解决：将不同职责分离到不同类中
示例：用户类分离为用户信息类和用户操作类
```

#### 开闭原则(OCP)
```
问题：修改现有代码来扩展功能
解决：通过继承和多态实现扩展
示例：使用策略模式支持多种算法
```

## 算法设计答题技巧

### 算法分析方法

#### 时间复杂度分析
**常见复杂度**
```
O(1)：常数时间 - 数组访问
O(log n)：对数时间 - 二分查找
O(n)：线性时间 - 遍历数组
O(n log n)：线性对数时间 - 归并排序
O(n²)：平方时间 - 冒泡排序
```

#### 空间复杂度分析
**分析要点**
- 递归调用栈的深度
- 额外数组的大小
- 临时变量的数量

### 常见算法模板

#### 排序算法
**快速排序**
```
1. 选择基准元素
2. 分区操作：小于基准的放左边，大于基准的放右边
3. 递归处理左右子数组
时间复杂度：平均O(n log n)，最坏O(n²)
```

**归并排序**
```
1. 分解：将数组分成两半
2. 解决：递归排序两个子数组
3. 合并：将两个有序数组合并
时间复杂度：O(n log n)
```

#### 查找算法
**二分查找**
```
1. 比较中间元素与目标值
2. 根据比较结果缩小查找范围
3. 重复直到找到或范围为空
时间复杂度：O(log n)
```

### 答题格式模板

#### 算法描述
```
1. 算法思想：简述算法的基本思路
2. 算法步骤：详细描述算法步骤
3. 伪代码：用伪代码表示算法
4. 复杂度分析：分析时间和空间复杂度
```

#### 伪代码规范
```
BEGIN
    输入：参数说明
    输出：返回值说明
    
    变量声明
    
    FOR/WHILE/IF 控制结构
        具体操作步骤
    END FOR/WHILE/IF
    
    RETURN 返回值
END
```

## 通用答题策略

### 审题技巧
1. **快速浏览**：先看题目类型和分值分布
2. **仔细阅读**：重点关注关键词和约束条件
3. **标记重点**：用笔标记重要信息
4. **理解需求**：确保理解题目要求

### 答题顺序
1. **先易后难**：从熟悉的题型开始
2. **高分优先**：优先完成分值高的题目
3. **时间控制**：为每道题设定时间限制
4. **预留检查**：留出10-15分钟检查时间

### 答题规范
1. **字迹清晰**：确保阅卷老师能看清
2. **图形规范**：使用标准的UML符号
3. **标注完整**：图形要有完整的标注
4. **逻辑清晰**：答案要有清晰的逻辑结构

### 得分技巧
1. **部分得分**：即使不完全正确也要写相关内容
2. **关键词得分**：确保包含题目要求的关键词
3. **步骤完整**：按照标准步骤完成答题
4. **检查验证**：完成后检查答案的合理性

### 时间管理
1. **分配时间**：根据分值合理分配时间
2. **控制节奏**：避免在某道题上花费过多时间
3. **灵活调整**：根据实际情况调整答题顺序
4. **最后冲刺**：利用最后时间完善答案
